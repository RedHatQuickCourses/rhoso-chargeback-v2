= Analyzing Chargeback Data and Consumption

Understanding and effectively analyzing chargeback data is crucial for cost management, resource optimization, and transparent billing within a cloud environment. RHOSO's Chargeback feature, powered by the Rating service (CloudKitty), provides the mechanisms to collect, rate, and report on resource consumption. This section will delve into how to access and interpret this valuable data to gain insights into usage patterns and associated costs.

== Retrieving Chargeback Data
The core of analyzing chargeback data lies in querying the aggregated consumption reports generated by the CloudKitty service. These reports provide a summary of resource usage over specified periods, allowing for granular analysis of costs.

CloudKitty exposes a REST API endpoint that allows you to retrieve these reports. The `v1/report/summary` endpoint is particularly useful for obtaining an overview of consumption, broken down by various parameters.

To access these reports, you typically use a `curl` command, authenticating with an appropriate token (e.g., `$TENANT_TOKEN` for a specific project).

=== Key Query Parameters

When querying chargeback reports, the following parameters are essential:

*   `begin`: Specifies the start timestamp for the reporting period (e.g., `2026-02-01T00:00:00`).
*   `end`: Specifies the end timestamp for the reporting period (e.g., `2026-03-01T00:00:00`).
*   `groupby`: This parameter is critical for breaking down the consumption data. It allows you to group the results by different criteria, such as resource type (`res_type`), tenant ID (`tenant_id`), or service type. The provided context specifically highlights `res_type`.

=== Hands-on Activity: Fetching a Basic Chargeback Summary Report

In this activity, you will use the CloudKitty API to retrieve a summary of resource consumption for a specific period, grouped by resource type.

==== Prerequisites

*   Access to an RHOSP environment with Chargeback enabled.
*   An authenticated token (`$TENANT_TOKEN`) for a project to query its consumption.
*   The CloudKitty API endpoint (e.g., `http://localhost:8888`).

==== Steps

.  **Set your Authentication Token and CloudKitty Endpoint**:
    Before running the `curl` command, ensure you have your tenant's authentication token set as an environment variable (e.g., `$TENANT_TOKEN`) and the correct CloudKitty API endpoint. For lab environments, `http://localhost:8888` is a common placeholder, but in a production environment, this would be the actual CloudKitty API URL.

    [source,bash]
    ----
    # Replace with your actual token
    export TENANT_TOKEN="YOUR_ACTUAL_AUTH_TOKEN"
    export CLOUDKITTY_API="http://localhost:8888" # Or your actual CloudKitty endpoint
    ----

.  **Execute the `curl` Command**:
    Use the following `curl` command to query a summary report. This example fetches data for February 2026, grouped by resource type.

    [source,bash]
    ----
    curl -X GET \
      -H "X-Auth-Token: $TENANT_TOKEN" \
      "${CLOUDKITTY_API}/v1/report/summary?begin=2026-02-01T00:00:00&end=2026-03-01T00:00:00&groupby=res_type"
    ----

.  **Review the Output**:
    The command will return a JSON object containing the summary data. Observe the structure and content of the response.

    [source,json]
    ----
    {
      "summary": [
        {
          "tenant_id": "MoD-project-uuid",
          "res_type": "ceilometer_cpu",
          "begin": "2026-02-01T00:00:00",
          "end": "2026-03-01T00:00:00",
          "rate": 125.50
        },
        {
          "tenant_id": "MoD-project-uuid",
          "res_type": "instance_hours",
          "begin": "2026-02-01T00:00:00",
          "end": "2026-03-01T00:00:00",
          "rate": 200.75
        },
        {
          "tenant_id": "MoD-project-uuid",
          "res_type": "volume_gb_hours",
          "begin": "2026-02-01T00:00:00",
          "end": "2026-03-01T00:00:00",
          "rate": 50.25
        }
      ]
    }
    ----

TIP: For readability, you can pipe the `curl` output to a JSON processor like `jq`: `curl ... | jq .`

== Interpreting Chargeback Reports

The JSON output from the `v1/report/summary` endpoint provides a clear, structured view of resource consumption and its associated "rate" or calculated units. Each object within the `"summary"` array represents a breakdown of consumption for a specific criterion.

Let's break down the components of a typical entry:

*   `tenant_id`: This unique identifier specifies the project or tenant for which the resource consumption is being reported. This is crucial for attributing costs to specific departments or users.
*   `res_type`: Indicates the type of resource being consumed (e.g., `ceilometer_cpu` for CPU usage, `instance_hours` for VM uptime, `volume_gb_hours` for storage consumption). This field is directly influenced by the `groupby` parameter in your query.
*   `begin` and `end`: These timestamps define the precise period over which the consumption was measured and aggregated. They align with the `begin` and `end` parameters provided in your `curl` query.
*   `rate`: This value represents the total *calculated units* for the specified `res_type` during the given period. It's the output of applying the defined rating rules to the collected metrics. While not a direct monetary cost, it's the quantified consumption that serves as the basis for financial billing.

[NOTE]
The `rate` field in CloudKitty's output represents the *calculated units* after applying rating rules, not necessarily the final monetary cost. CloudKitty provides the crucial link between usage and these calculated units, which are then typically funneled into an organization's financial software for final cost accountability and invoicing.

== Analyzing Consumption Patterns

Analyzing chargeback data goes beyond just generating reports; it involves extracting meaningful insights to inform decision-making.

*   **Identifying High-Consumption Areas**: By grouping data by `res_type` (resource type), you can quickly identify which resources (e.g., CPU, storage, network) are being heavily utilized and contributing significantly to the `rate`. This helps in resource planning and optimization.
*   **Tenant-Level Consumption**: If you were to use `groupby=tenant_id`, you could see the total `rate` for each tenant, enabling you to understand which projects or departments are consuming the most resources. This is essential for cross-charging.
*   **Trend Analysis**: By running reports over different time periods (e.g., weekly, monthly, quarterly) and comparing the `rate` values, you can identify consumption trends. Are certain resources increasing in usage? Are there seasonal peaks? This helps in forecasting and budget allocation.
*   **Performance vs. Cost**: Combining chargeback data with performance metrics can help you understand if high-cost resources are delivering equivalent value or if there are opportunities for efficiency improvements (e.g., rightsizing VMs).
*   **Chargeback Accuracy Verification**: Regular analysis of reports helps in verifying that the rating rules are being applied correctly and that the calculated `rate` accurately reflects the intended consumption.

For deeper analysis, you can adjust the `groupby` parameter in your `curl` command. For instance, while not explicitly shown in the context, a common practice would be to group by `tenant_id` to see the total consumption per project:

[source,bash]
----
curl -X GET \
  -H "X-Auth-Token: $TENANT_TOKEN" \
  "${CLOUDKITTY_API}/v1/report/summary?begin=2026-02-01T00:00:00&end=2026-03-01T00:00:00&groupby=tenant_id"
----

This would provide a summary of the `rate` accumulated by each distinct `tenant_id` within the specified period, offering a different perspective on overall consumption.

== Integrating with Financial Software

While CloudKitty excels at collecting metrics, applying rating rules, and generating structured consumption reports, it is *not* a full-fledged billing solution. Its primary role is to serve as the "crucial puzzle piece linking usage to cost" by providing the "structured, aggregated JSON data."

The final step in the chargeback process is to funnel this data directly into your enterprise's broader financial software, such as an accounting system or an internal billing platform. This external system then takes the `rate` (calculated units) from CloudKitty's reports and applies specific monetary values (e.g., "$5 per unit of `ceilometer_cpu`") to generate actual invoices and ensure cost accountability.

By systematically analyzing the chargeback data and integrating it with financial systems, organizations can achieve:

*   **Transparent Cost Allocation**: Clearly show how resource consumption translates into costs for each project or department.
*   **Improved Budgeting and Forecasting**: Use historical consumption data to make more accurate predictions and allocate budgets more effectively.
*   **Resource Optimization**: Identify underutilized or over-provisioned resources by linking high costs to low actual usage.
*   **Enhanced Accountability**: Foster a culture of cost awareness among cloud users, encouraging efficient resource usage.

In essence, the cycle of `Set Rating Rules → Collect Metrics → Generate Rating Reports` culminates in the analysis of these reports, which then feeds into the broader financial ecosystem to close the loop between infrastructure consumption and financial accountability.
