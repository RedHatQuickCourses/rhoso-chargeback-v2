Verifying successful Chargeback activation

Verifying Successful Chargeback Activation

After enabling the Chargeback feature through the Telemetry Operator, it is crucial to verify that all necessary components are correctly deployed and operational. This verification process ensures that the Rating service (CloudKitty) and its dependencies are running as expected, allowing for accurate chargeback and rating capabilities within your RHOSO cloud environment. A successful activation means that the underlying infrastructure is ready to process resource consumption data and generate billing information.

Detailed Technical Explanation

The Chargeback functionality in RHOSO heavily relies on the Telemetry Operator, which manages services like CloudKitty for rating. Before CloudKitty can process rating definitions and generate reports, its dependencies, particularly the MetricStorage, must be fully operational. The MetricStorage Custom Resource (CR) ensures that a robust backend is available for storing metric data, which CloudKitty consumes for its calculations. Furthermore, the OpenShift Cloud Platform (OSCP) Controlplane CR explicitly dictates the enablement of telemetry and cloudkitty within your RHOSO deployment. Verifying these configurations is paramount to confirming a successful Chargeback activation.

Failure to verify these steps can lead to:

  • Chargeback features not appearing in the RHOSO user interface.

  • Inaccurate or missing chargeback data.

  • Errors when trying to define rating rules or generate reports.

Hands-on Activity: Verifying Chargeback Activation

This hands-on activity guides you through the process of verifying the successful activation of the Chargeback feature using oc commands.

Prerequisites

  • You have access to an OpenShift cluster where RHOSO is deployed.

  • You are logged in to the OpenShift cluster using oc login.

  • You have the jq utility installed for parsing JSON output.

Step 1: Verify MetricStorage Status

The MetricStorage Custom Resource is fundamental for storing telemetry data that CloudKitty processes. A "Setup complete" status indicates that the storage backend is ready.

Verification Steps

  1. Execute the following command to check the MetricStorage CR status:

    oc get metricstorage

  2. Expected Output:

    NAME             NETWORKATTACHMENTS   STATUS   MESSAGE
metric-storage                        True     Setup complete

    If the MESSAGE column shows "Setup complete", the MetricStorage is correctly configured.

  3. Troubleshooting MetricStorage Status:

    If the MESSAGE is not "Setup complete", it indicates an issue with the metric storage setup. Pay attention to the message itself, as it might provide clues. For more detailed status conditions, use the following command:

    oc get metricstorage metric-storage -ojsonpath --template "{.status.conditions}" | jq

    This command provides a JSON array of conditions, including lastTransitionTime and potentially more descriptive reason and message fields, which can help diagnose the problem. For instance, you might see conditions related to pending deployments, Persistent Volume Claim (PVC) issues, or network misconfigurations. Resolve any issues indicated by these conditions before proceeding.

Step 2: Verify Telemetry and CloudKitty Enablement in Controlplane

The Controlplane Custom Resource for RHOSO (oscp) explicitly enables or disables core services. For Chargeback, both telemetry and cloudkitty must be enabled.

Verification Steps

  1. Check if telemetry is enabled in the Controlplane CR:

    oc get oscp -ojsonpath --template "{.items[0].spec.telemetry.enabled}" | jq

  2. Expected Output:

    true

  3. Check if cloudkitty is enabled within the telemetry section of the Controlplane CR:

    oc get oscp -ojsonpath --template "{.items[0].spec.telemetry.template.cloudkitty.enabled}" | jq

  4. Expected Output:

    true

  5. Troubleshooting Telemetry and CloudKitty Enablement:

    If either command returns false instead of true, it means that telemetry or cloudkitty (or both) are not enabled in your Controlplane configuration. You will need to edit the Controlplane CR to switch these fields to true.

  6. To edit the Controlplane CR, use the following command:

    oc edit oscp

  7. Navigate to the spec.telemetry.enabled and spec.telemetry.template.cloudkitty.enabled fields and change their values to true. Save the CR.

    apiVersion: xxx/v1beta1
kind: OpenStackControlPlane
metadata:
  name: openstack-controlplane # Or your specific controlplane name
spec:
  telemetry:
    enabled: true <1>
    template:
      cloudkitty:
        enabled: true <2>
    1 Ensure this is true to enable the Telemetry Operator.
    2 Ensure this is true to enable CloudKitty (the Rating service).

    After editing and saving the CR, the Telemetry Operator will reconcile the changes, deploying or reconfiguring the necessary components. You might need to wait a few minutes for the changes to propagate and for the associated pods to restart or come online. Re-run the verification commands to confirm the true status.

Summary

By successfully completing these verification steps, you have confirmed that the underlying infrastructure for RHOSO Chargeback is correctly configured and operational. This lays the foundation for defining rating rules, managing services, and generating accurate chargeback reports, enabling effective cost management within your RHOSP environment.
