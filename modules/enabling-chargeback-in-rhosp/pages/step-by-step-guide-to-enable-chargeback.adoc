#  Step-by-step guide to enable Chargeback

= Enabling Chargeback in RHOSP
:navtitle: Enable Chargeback
:page-aliases: chargeback:enabling-chargeback.adoc

In this section, you will learn how to enable the Chargeback feature within Red Hat OpenStack Platform (RHOSP) by activating the underlying Rating service. This crucial step lays the foundation for transparent cost recovery and resource optimization across your cloud environment.

[[_prerequisites_for_chargeback_deployment]]
== Prerequisites for Chargeback Deployment in RHOSP

Before proceeding with enabling Chargeback, ensure the following foundational elements are in place:

*   **RHOSP Environment:** A fully deployed and operational Red Hat OpenStack Platform environment.
*   **Telemetry Operator:** The Telemetry Operator must be deployed and running in your RHOSP environment (or the OpenShift cluster managing RHOSP components). As per the context, the Chargeback functionality, powered by CloudKitty, is managed and deployed through this operator.
*   **Administrative Access:** You must have administrative access to the RHOSP environment and the OpenShift cluster (if applicable) where the Telemetry Operator resides, to modify its configuration.

[[_step_by_step_guide_to_enable_chargeback]]
== Step-by-Step Guide to Enable Chargeback

The Chargeback feature in RHOSO is primarily provided by the *Rating service*, also known as `cloudkitty`. As highlighted in the provided context, you enable `cloudkitty` within the Telemetry Operator to provide chargeback and rating capabilities.

The process involves modifying the Telemetry Operator's Custom Resource (CR) to activate the `cloudkitty` component. While the specific configuration details can vary slightly depending on the exact version and deployment method of the Telemetry Operator, the general workflow involves accessing your OpenShift cluster and updating the relevant Custom Resource.

.Conceptual Steps to Enable CloudKitty via Telemetry Operator
. *Access the OpenShift Cluster:* Log in to your OpenShift cluster where the Telemetry Operator is deployed. This is typically done using the `oc` CLI tool.
+
[,bash]
----
oc login --token=<your_token> --server=<your_api_server>
----
. *Identify the Telemetry Operator's Custom Resource (CR):* The Telemetry Operator manages its components, including `cloudkitty`, through a Custom Resource. You'll need to locate and inspect this CR. The exact name and kind might vary depending on your specific RHOSP deployment, but it's often related to `Telemetry` or `OpenStackTelemetry`.
+
[,bash]
----
oc get <telemetry-operator-cr-kind> -n <openstack-namespace>
----
+
[NOTE]
Consult your specific RHOSP and Telemetry Operator documentation to find the correct `kind` and name of the Telemetry Operator's custom resource. This will provide the precise target for your configuration changes.
. *Edit the Custom Resource to Enable CloudKitty:* Edit the identified Custom Resource to enable the `cloudkitty` component. This typically involves modifying the YAML definition of the CR. You might need to add a dedicated section for `cloudkitty`, set an `enabled` flag to `true`, or ensure `cloudkitty` is explicitly listed among the enabled services.
+
For example, you might add or modify a section similar to this within the CR's `spec`:
+
[,yaml]
----
apiVersion: telemetry.openstack.redhat.com/v1beta1
kind: OpenStackTelemetry
metadata:
  name: openstack-telemetry-default # Or your specific CR name
spec:
  # ... other telemetry configurations ...
  cloudkitty:
    enabled: true
    # Further CloudKitty-specific configurations might be required here,
    # such as API replicas, database settings, or integration parameters.
----
+
[WARNING]
Always back up your Custom Resource definition before making any modifications. Incorrect changes can disrupt your Telemetry services and impact the stability of your RHOSP environment.
. *Apply the Changes:* Save the modified Custom Resource YAML and apply it using the `oc apply` command. The Telemetry Operator will detect this change and initiate the reconciliation process, deploying or configuring `cloudkitty` accordingly.
+
[,bash]
----
oc apply -f <your-modified-telemetry-cr.yaml>
----
. *Monitor Deployment:* Monitor the deployment of the `cloudkitty` pods and services to ensure they start correctly. You can check the status of the Telemetry Operator and its managed pods within the relevant OpenShift namespace.
+
[,bash]
----
oc get pods -n <openstack-namespace> | grep cloudkitty
oc logs -f <cloudkitty-api-pod-name> -n <openstack-namespace> # To check specific pod logs
----

[[_configuration_parameters_and_settings]]
== Configuration Parameters and Settings

Once `cloudkitty` is enabled, there are several configuration parameters and settings that you may need to adjust based on your specific chargeback requirements and scale. These often include:

*   **API Replicas:** Define the number of `cloudkitty` API instances to ensure high availability and efficiently handle the expected load.
*   **Database Backend:** Configure the connection details for the database `cloudkitty` uses to store rating data, rules, and generated reports.
*   **Collector Integration:** Ensure `cloudkitty` is correctly configured to consume metrics from various OpenStack services (such as Nova for compute, Cinder for block storage, Neutron for networking, and Glance for images) via the integrated metric collection agents (e.g., Ceilometer).
*   **Rating Service Parameters:** Fine-tune parameters related to how `cloudkitty` processes metrics and applies rules, such as polling intervals, data aggregation settings, and retention policies.

These settings are typically defined directly within the `OpenStackTelemetry` Custom Resource itself, under the `cloudkitty` section, or in associated configuration maps that the Telemetry Operator manages. Refer to the official Red Hat documentation for your specific RHOSP version for detailed configuration options.

[[_verifying_successful_chargeback_activation]]
== Verifying Successful Chargeback Activation

After enabling and configuring `cloudkitty`, it is crucial to verify that the Chargeback functionality is active and working correctly.

.Verification Steps
. *Check CloudKitty Pod Status:* Confirm that the `cloudkitty` pods are running and healthy within the designated namespace.
+
[,bash]
----
oc get pods -n <openstack-namespace> -l app=cloudkitty
----
+
All `cloudkitty` pods (e.g., `cloudkitty-api`, `cloudkitty-processor`) should be in a `Running` or `Completed` state.
. *Inspect CloudKitty Service Endpoints:* Verify that the `cloudkitty` API service is accessible within the cluster.
+
[,bash]
----
oc get svc -n <openstack-namespace> | grep cloudkitty
----
. *Review Telemetry Operator Status:* Check the status of the `OpenStackTelemetry` Custom Resource to ensure `cloudkitty` is reported as healthy and deployed without errors.
+
[,bash]
----
oc get OpenStackTelemetry <your-cr-name> -o yaml -n <openstack-namespace>
----
+
Look for conditions or status fields within the YAML output that indicate `cloudkitty`'s operational state and successful deployment.
. *Perform a Basic Rating Rule Test (Future Lab):* While this section focuses on enabling, a definitive verification involves creating a simple rating rule and observing if `cloudkitty` successfully processes metrics and generates initial reports. This practical application will be covered in detail in the "Hands-on Lab: Implementing Your First Chargeback Rating."

By following these steps, you can successfully enable the core Chargeback capabilities in your RHOSP environment, powered by the `cloudkitty` Rating service. This activation is the prerequisite for configuring rating rules and leveraging detailed resource consumption analysis for enhanced cost transparency.
