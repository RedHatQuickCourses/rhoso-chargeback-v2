Defining a basic rating rule for a service

Defining a Basic Rating Rule for a Service

In the previous lab, we prepared our environment for RHOSO Chargeback. Now, we’ll proceed with a crucial step: defining our very first rating rule. This fundamental configuration allows us to assign costs to specific services or resources consumed within our OpenStack environment, enabling accurate cost attribution and financial transparency.

Understanding Rating Rules for Services

A rating rule in RHOSO Chargeback is a configuration that dictates how a specific service or resource consumption metric is translated into a financial charge. When we define a rating rule for a service, we are essentially establishing a policy that tells RHOSO: "When X service or a specific attribute of X service is consumed, apply Y cost."

The core principle, as outlined in the provided context, involves "creating mappings from a field or service that triggers the application of the rating rule." This means we identify a particular attribute (a 'field') or a specific service that RHOSO monitors, and then we associate a monetary value with its usage.

For instance, the context explicitly states, "you can add a rating to a specific value of flavor on an instance." This capability is vital for implementing differentiated pricing based on the specifications of virtual machines users provision. Similarly, you "Create a mapping on a service, or metric name, to associate a rating directly with that metric," allowing for broad or granular cost application.

Key components that are essential when defining a basic rating rule:

  • Service/Metric Identifier: This is the specific service, resource, or metric that RHOSO observes, which should incur a charge. Examples include compute flavors, storage volume types, network egress traffic, etc. The mapping directly associates the rating with this identifier.

  • Trigger/Condition (Mapping): This specifies when the rule applies. For example, if the field is flavor and its value is m1.small, the rule triggers.

  • Rating Value/Rate: This is the actual monetary amount to be charged. It includes the amount, currency, and the interval (e.g., per hour, per GB, per API call).

  • Effective Date: As per the context, this "Specifies the date from which the rating applies." This effective_from timestamp is critical for managing pricing changes over time and ensuring historical accuracy.

By precisely defining these components, we ensure that all tracked resource consumption is accurately translated into financial data for chargeback reports, providing clear insights into operational costs.

Hands-on Lab: Defining a Basic Rating Rule for a Compute Flavor

In this hands-on lab, we will walk through the process of defining a simple rating rule to charge for a specific OpenStack compute flavor. Our scenario involves applying a charge of $0.05 per hour for instances that utilize the m1.small flavor.

The specific commands, configuration file formats, or UI elements might vary depending on your RHOSO deployment version and any custom integrations. This lab provides a conceptual understanding and illustrative steps to guide you through the process.

Lab Objective

  • Create a basic chargeback rating rule targeting the m1.small compute flavor.

  • Configure a per-hour rate for instances using this specific flavor.

  • Set an appropriate effective start date for the rule’s application.

Steps to Define the Rating Rule

Follow these steps to configure your first basic chargeback rating rule within RHOSO.

  1. Identify the Metric or Service to Rate

    Our objective is to rate the m1.small compute flavor. RHOSO continuously collects metrics related to OpenStack instances, including their associated flavor. We need to tell RHOSO to look for this specific flavor value.

  2. Access the Chargeback Configuration Interface

    Typically, chargeback rules are configured either through the RHOSO web-based user interface or via a dedicated command-line interface (CLI) tool. For this lab, we will use a conceptual YAML-based configuration structure to illustrate the rule definition, which is a common approach for infrastructure-as-code.

  3. Define the Rating Rule Structure

    We need to create a mapping that precisely links the m1.small flavor to its specific hourly rate. As the context indicates, this involves mapping a field (flavor) to a specific value (m1.small) within a service_type (OpenStack Compute instances).

    Create a file, for example, chargeback_rules.yaml, with the following content:

    # chargeback_rules.yaml
rating_rules:
  - name: "Small_Flavor_Hourly_Charge"
    description: "Hourly charge for m1.small compute instances"
    applies_to:
      service_type: "OpenStack Compute" # Targets metrics from OpenStack Compute service
      field: "flavor_name"              # The specific field representing the flavor
      value: "m1.small"                 # The value of the field this rule applies to
    rate:
      amount: 0.05
      unit: "USD"
      interval: "hourly"                # The rate is applied per hour
    effective_from: "2023-01-01T00:00:00Z" # "Specifies the date from which the rating applies."
    # Additional optional fields could include 'valid_until', 'currency_code', etc.

    The effective_from field is critical. It ensures that this rating rule is only applied to usage data recorded from the specified date and time onwards, providing essential flexibility for managing pricing changes and historical data.

  4. Apply the Rating Rule

    Once your rule definition file (e.g., chargeback_rules.yaml) is prepared, you would typically apply it using a management command via the RHOSO CLI or by uploading it through the RHOSO UI.

    # Conceptual command to apply the rating rule definition
rhoso-cli chargeback rule apply -f chargeback_rules.yaml

    Always thoroughly review your rating rule definitions before applying them, especially in a production environment. Incorrectly configured rules can lead to inaccurate charge calculations and misleading chargeback reports.

  5. Verify the Rule’s Configuration (Optional but Recommended)

    After applying the rule, it’s good practice to verify its successful configuration. The context mentions, "You can see which metrics are configured for rating with [a command/UI section]". Use a conceptual command to list active rating rules.

    # Conceptual command to list active rating rules
rhoso-cli chargeback rule list

    Examine the output to confirm that Small_Flavor_Hourly_Charge appears in the list of active rules, indicating its successful deployment.

By completing these steps, you have successfully defined your first basic rating rule in RHOSO Chargeback. This rule will now begin attributing costs to the m1.small compute flavor based on its hourly consumption from the specified effective date, laying a solid foundation for more intricate chargeback scenarios.
