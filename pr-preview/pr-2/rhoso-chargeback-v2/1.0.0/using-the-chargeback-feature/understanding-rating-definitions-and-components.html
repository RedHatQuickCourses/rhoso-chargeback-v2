Understanding rating definitions and components

Understanding Rating Definitions and Components

Chargeback systems, such as the one offered by RHOSO, are instrumental in transforming raw resource consumption data into actionable financial insights. At the heart of this transformation lie rating definitions or rating rules. These rules provide the intelligence needed to accurately attribute costs, foster accountability, and drive efficient resource utilization across an organization. This section will demystify rating definitions and break down their essential components, providing a foundational understanding for effective Chargeback implementation.

What is a Rating Definition?

A rating definition, often synonymously referred to as a rating rule, is a meticulously structured instruction set that dictates how specific resource or service consumption within an environment should be financially valued. It’s the blueprint that allows RHOSO’s Chargeback engine to translate observable technical metrics into quantifiable costs, making infrastructure consumption transparent and auditable.

As the provided context states, "You create rating rules by creating mappings from a field or service that triggers the application of the rating rule." This emphasizes the core function of a rating definition: establishing a clear link between a specific condition or event related to a resource and the financial charge associated with it. For example, if your organization aims to charge for the usage of virtual machines, a rating definition would specify that for every hour a VM is actively running, a certain monetary value should be applied to the consuming project or tenant.

The primary goal of a well-crafted rating definition is to ensure that the charges levied are fair, predictable, transparent, and accurately reflect the actual cost incurred by the consumption of resources. This not only aids in cost recovery but also serves as a powerful incentive for users to optimize their resource footprint.

Key Components of a Rating Rule

To effectively configure and manage Chargeback in RHOSO, it is crucial to understand the individual building blocks that constitute a comprehensive rating rule. While implementations may vary in granular detail, the following core components are universally present in RHOSO’s Chargeback system:

Trigger/Condition

The trigger or condition is the logical expression that determines when a particular rating rule should become active and what specific resource consumption or attribute causes the charge to be applied. It’s the "if" clause of your rating logic.

As highlighted in the context, "You create rating rules by creating mappings from a field or service that triggers the application of the rating rule." This mapping can be based on several factors:

  • Service Type: The specific type of infrastructure service being consumed (e.g., Compute, Block Storage, Object Storage, Network).

  • Resource Attributes: Specific properties or characteristics of a resource. The context offers a clear example: "For example, you can add a rating to a specific value of flavor on an instance." This allows for differentiated pricing, where a m1.large VM instance might have a different base rate than an m1.small instance.

  • Events: Specific actions or lifecycle events (e.g., creation of a resource, deletion, snapshotting).

Effectively defining triggers allows administrators to apply different pricing models based on varying service levels, performance tiers, or operational costs.

Metric

The metric represents the quantifiable unit of consumption upon which the charge is based. It defines what specific aspect of resource usage is being measured and charged for.

The context refers to metrics indirectly by stating, "You can see which metrics are configured for rating with If something should be rated, but it’s missing from the list, you can use the following procedure for configuring new rating rules." This implies that metrics are the core data points that the rating engine consumes.

Examples of common metrics include:

  • Compute: CPU-hours, RAM-GB-hours, VM-hours (the duration a VM is active).

  • Storage: GB-months (average storage consumed over a month), IOPS (Input/Output Operations Per Second).

  • Network: GB transferred (data ingress/egress), network port hours.

RHOSO leverages its extensive observability capabilities to collect a wide array of metrics, providing the raw data necessary for granular and accurate chargeback calculations.

Rate (Price)

The rate, or price, is the monetary value assigned to a single unit of the chosen metric. This is the ultimate financial quantification of consumption.

  • Example: If the metric is "CPU-hour," the rate might be $0.05 per CPU-hour. If it’s "GB-month of storage," the rate could be $0.01 per GB-month.

  • Rates can be configured as:

  • Fixed: A constant price per unit.

  • Tiered: Different rates apply based on consumption bands (e.g., first 100 GB are $X/GB, next 200 GB are $Y/GB).

  • Volume-based: Rates decrease as total consumption increases.

Periodicity/Applicability Date

This component defines the timeframe during which a rating rule is active and applicable. It’s crucial for managing versioning of pricing policies and historical data.

The context explicitly highlights this with the statement: "Specifies the date from which the rating applies."

  • Start Date: The specific date and time from which the rating rule becomes active and starts influencing charge calculations.

  • End Date (Optional): An expiry date and time when the rule ceases to be active. This is useful for temporary promotions or planned pricing changes.

  • Billing Period: While not a direct component of an individual rating rule, the overall Chargeback system will typically aggregate charges over predefined billing periods (e.g., monthly, quarterly, annually) for reporting and invoicing.

Target/Scope

The target or scope dictates to whom or what specific entities a rating rule applies. This provides the flexibility to implement varying pricing models for different internal customers or departments.

  • Project/Tenant: A common approach is to apply rules to specific OpenStack projects or tenants, allowing different departments to have custom pricing.

  • User Group: Rules could be tailored for different user groups within an organization.

  • Geographical Region/Availability Zone: If your infrastructure spans multiple regions or availability zones, different rates might apply based on location-specific costs.

How Rating Rules Work in Practice

When resources are provisioned and consumed within RHOSP, the RHOSO Chargeback engine continuously monitors their usage and associated metrics. During a chargeback cycle, it queries the active rating rules. For each instance of resource consumption, it first identifies if any rating rule’s trigger/condition is met. If a match is found, it then uses the defined metric to quantify the usage. This usage is then multiplied by the specified rate (price) and attributed to the appropriate target/scope. All these calculations adhere to the periodicity/applicability date defined for the rule.

This systematic process ensures that all chargeable resource consumption is accurately recorded and aggregated, forming the basis for comprehensive chargeback reports and invoices. The meticulous combination of these components allows organizations to build highly flexible, transparent, and accurate chargeback models that align with their financial governance and operational objectives.
