RHOSO architectural overview

RHOSO Architectural Overview

This section provides a detailed look into the architectural design of Red Hat OpenStack Services on OpenShift (RHOSO), with a particular focus on its observability components. Understanding this architecture is crucial for effectively managing, monitoring, and troubleshooting your RHOSO environment.

Introduction to RHOSO Architecture

Red Hat OpenStack Services on OpenShift (RHOSO) represents a significant evolution in deploying OpenStack, where the entire OpenStack platform runs as a workload on top of Red Hat OpenShift Container Platform (RHOCP). This fundamental shift necessitated a re-evaluation of how core services, especially observability, are implemented.

The RHOSO observability architecture is designed to provide comprehensive insight into your environment by integrating services from both the underlying Red Hat OpenShift Container Platform and the OpenStack Compute nodes. It serves as the primary observability solution for the latest OpenStack Platform (OSP) releases, specifically replacing the Service Telemetry Framework in RHOSO 18 and beyond.

Core Architectural Components

The RHOSO observability architecture leverages a hybrid approach, combining established OpenShift tooling with native OpenStack telemetry solutions. This integration ensures robust metric collection, logging, and alerting capabilities across the entire RHOSO stack.

OpenShift-Native Observability Components

Within RHOCP, RHOSO utilizes several core OpenShift observability services:

  • Red Hat OpenShift Observability (COO): This refers to the suite of tools and operators within OpenShift that manage the collection, storage, and visualization of cluster metrics, logs, and alerts. For RHOSO, this forms the backbone of the monitoring infrastructure.

  • Cluster Logging Operator (CLO): Responsible for managing the lifecycle and configuration of logging components within OpenShift, including the collection, forwarding, and storage of logs from RHOSO services running on RHOCP.

  • OpenShift Console UI: Provides a unified graphical interface for visualizing metrics, logs, and alerts collected by the OpenShift observability stack, offering a central point for operational insight into RHOSO.

  • Telemetry Operator: This operator acts as a crucial bridge, integrating OpenStack-specific telemetry data into the OpenShift observability ecosystem. It ensures that metrics and logs from OpenStack services are properly collected and exposed to the OpenShift monitoring tools.

OpenStack-Native Telemetry Components

While leveraging OpenShift’s strengths, RHOSO also retains key OpenStack telemetry solutions that have been proven and familiar to OpenStack administrators:

  • Ceilometer: The traditional OpenStack telemetry service responsible for collecting various types of data (performance, events, usage) from OpenStack components. Ceilometer continues to play a role in gathering OpenStack-specific metrics.

  • AODH (Alarming-on-Demand-for-Heat): The OpenStack alarming service, which processes metrics from Ceilometer and other sources to trigger alerts based on defined rules.

The combination of COO + telemetry-operator encapsulates the primary mechanism for RHOSO Observability, demonstrating how OpenShift tools are extended to cover OpenStack workloads.

Deployment and Scope

The services comprising the RHOSO observability architecture are deployed across different layers of your infrastructure:

  • Within Red Hat OpenShift Container Platform (RHOCP): Services like COO, CLO, and the Telemetry Operator run directly within the OpenShift cluster, managing the overarching observability framework.

  • On Compute Nodes: Services on your OpenStack Compute nodes are responsible for providing node-specific metrics, logs, and alerts, ensuring full coverage of your data plane.

By default, RHOSO observability logging is enabled on the control plane components. However, for comprehensive monitoring, you can extend logging to the data plane.

Benefits of the RHOSO Observability Architecture

This hybrid architectural approach offers several significant advantages:

  • Leveraging Proven OpenShift Tooling: It utilizes tools and operators within OpenShift that are robust, scalable, and well-understood by OpenShift customers.

  • Familiarity for OpenShift Administrators: Operators and UI elements familiar to OpenShift users contribute to a reduced learning curve and smoother operations.

  • Enhanced Integration with OpenShift Ecosystem: Tighter integration allows for a more unified operational experience, where OpenStack metrics and logs are seamlessly part of the broader OpenShift monitoring landscape.

  • Retaining OpenStack Administrator Solutions: By keeping components like Ceilometer and AODH, it ensures that administrators familiar with traditional OpenStack telemetry can continue to leverage their existing knowledge and tools.

  • Centralized Logging and Monitoring: Provides a single pane of glass for both OpenShift infrastructure and OpenStack workloads, simplifying troubleshooting and performance analysis.

Enabling Data Plane Logging (Hands-on Activity)

While control plane logging is enabled by default, extending observability to the data plane is a critical step for comprehensive monitoring of your RHOSO environment. This involves configuring your OpenStackDataPlaneNodeSet Custom Resources (CRs).

Step-by-Step Guide: Enable Data Plane Logging

To enable Red Hat OpenStack Services on OpenShift (RHOSO) observability logging on the data plane, you must add the OpenStackDataPlaneService logging service to the services list of each OpenStackDataPlaneNodeSet custom resource (CR) defined for your data plane.

Prerequisites

Ensure you have oc (OpenShift CLI) configured and access to your RHOCP cluster where RHOSO is deployed.

Procedure

  1. Identify your OpenStackDataPlaneNodeSet CRs: First, list your OpenStackDataPlaneNodeSet resources to identify which ones manage your data plane nodes. bash oc get OpenStackDataPlaneNodeSet -n openstack

  2. Edit the OpenStackDataPlaneNodeSet CR: For each data plane node set where you want to enable logging, edit its CR. Replace example-dataplanenodeset with the actual name of your node set. bash oc edit OpenStackDataPlaneNodeSet example-dataplanenodeset -n openstack

  3. Add the logging service: Locate the spec.services section within the CR. Add logging to the list of services.

    [source,yaml]
----
# ... (other parts of the CR)
spec:
  services:
    - repo-setup
    - sshd
    - ovn
    - neutron_ovs
    - nova
    - nova_libvirt
    - logging (1)
# ... (other parts of the CR)
----
<1> Add `logging` to enable data plane observability logging.

  4. Save the changes: Save and exit the editor. The OpenStackDataPlane Operator will detect the change and reconcile the OpenStackDataPlaneNodeSet CR, deploying the necessary logging components to the specified data plane nodes.

  5. Verify successful activation: Monitor the status of your OpenStackDataPlaneNodeSet CR and the pods on your data plane nodes. You should observe new logging-related pods or agents being deployed and running, indicating successful activation. You can also verify by checking logs from a data plane compute node using OpenShift’s logging tools (e.g., in the OpenShift console or via oc logs).

This architectural overview provides the foundation for understanding how RHOSO operates and how its observability is designed, setting the stage for deeper dives into specific features like Chargeback.
