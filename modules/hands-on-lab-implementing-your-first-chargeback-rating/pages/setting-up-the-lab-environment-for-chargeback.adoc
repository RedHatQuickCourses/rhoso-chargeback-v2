# Setting up the lab environment for Chargeback

In this unit, you will prepare your Red Hat OpenStack Services on OpenShift (RHOSO) environment for chargeback by installing the required dependencies and enabling the Rating service (CloudKitty). You will use Ansible playbooks to automate the setup, while understanding what each step accomplishes based on the concepts covered in the xref:../enable-rating-in-rhoso/prerequisites-for-rating-enablement-in-rhoso.adoc[prerequisites] and xref:../enable-rating-in-rhoso/how-to-enable-rating.adoc[enablement] theory modules.

== Prerequisites

Before you begin, ensure you have the following:

* Access to an RHOSO cluster with administrator privileges.
* The OpenShift CLI (`oc`) installed and authenticated against the target cluster.
* A valid `KUBECONFIG` pointing to the target cluster.
* Ansible installed on your workstation.

== Connect to the Utility Machine

All lab tasks are performed from the utility machine. From your workstation, connect to it via SSH:

[source,bash]
----
[student@workstation ~]$ ssh utility
----

Once connected, install Ansible Core:

[source,bash]
----
[student@utility ~]$ sudo dnf install ansible-core
----

You will run all subsequent commands from this machine.

== Clone the Playbooks Repository

All the automation scripts for this lab are available in a dedicated Git repository. Clone it to your workstation:

[source,bash]
----
git clone https://github.com/opentraining-labs-customizations/rhoso-chargeback.git
cd rhoso-chargeback
----

The repository contains four directories, each with an Ansible playbook for a specific stage of the lab:

* `pre-deploy/` -- Installs Loki and MinIO, required by CloudKitty for rated data storage.
* `deploy/` -- Updates the OpenStack Control Plane to enable Telemetry, Ceilometer, MetricStorage, and CloudKitty.
* `create-ck-rating/` -- Automates the creation of a hashmap rating rule (used in the next unit).
* `validation/` -- Validates that the rating configuration was applied correctly (used in the final unit).

== Install Loki and MinIO

As covered in the xref:../enable-rating-in-rhoso/prerequisites-for-rating-enablement-in-rhoso.adoc[prerequisites module], CloudKitty requires a Loki instance for storing rated data, backed by S3-compatible storage. The `pre-deploy` playbook sets this up for you.

Run the playbook:

[source,bash]
----
ansible-playbook pre-deploy/cloudkitty-pre_deploy-install_loki.yml
----

This playbook performs the following actions:

. Waits for the OpenShift Route API to become available.
. Applies Kubernetes manifests that create:
* The `openshift-operators-redhat` and `minio-dev` namespaces.
* A Loki Operator subscription (from the `stable-6.4` channel).
* A MinIO pod with persistent storage (10Gi) for S3-compatible object storage.
* A Secret named `cloudkitty-loki-s3` in the `openstack` namespace containing MinIO credentials.
. Waits for the Loki Operator to reach `Succeeded` status (up to 5 minutes).

[NOTE]
====
The playbook references files relative to the repository root. If you are running it from a different directory, override the path with:

[source,bash]
----
ansible-playbook \
  -e "ansible_user_dir=/path/to/rhoso-chargeback" \
  pre-deploy/cloudkitty-pre_deploy-install_loki.yml
----
====

== Enable Telemetry, Ceilometer, MetricStorage, and CloudKitty

With the storage dependencies in place, you can now enable the Rating service itself. As described in the xref:../enable-rating-in-rhoso/how-to-enable-rating.adoc[enablement module], this involves modifying the `OpenStackControlPlane` Custom Resource to activate several components.

Run the deploy playbook:

[source,bash]
----
ansible-playbook deploy/update-deployment-for-ck.yml
----

This playbook performs the following actions:

. Adds a `CloudKittyPassword` entry to the `osp-secret` Secret.
. Activates the Telemetry service in the `OpenStackControlPlane` CR.
. Activates Ceilometer in the `OpenStackControlPlane` CR.
. Activates MetricStorage with `pvcStorageClass: nfs-storage`.
. Activates CloudKitty in the `OpenStackControlPlane` CR.
. Waits for the `OpenStackControlPlane` to become ready (up to 30 minutes).

[IMPORTANT]
====
This step may take up to 30 minutes to complete while the operator reconciles all changes and deploys the new components. Do not interrupt the playbook while it is waiting.
====

== Verify the Deployment

After the playbook completes successfully, verify that CloudKitty is running.

. Check that the CloudKitty pods are in `Running` state:
+
[source,bash]
----
oc get pods -n openstack | grep cloudkitty
----
+
You should see `cloudkitty-api` and `cloudkitty-processor` pods running.

. Confirm the CloudKitty CR reports a `Ready` condition:
+
[source,bash]
----
oc get -n openstack CloudKitty
----

. Verify that Ceilometer and MetricStorage are also ready:
+
[source,bash]
----
oc get ceilometer -n openstack
oc get metricstorage -n openstack
----

Once all components report a ready state, your environment is fully prepared for rating. In the next unit, you will create your first rating rule.
