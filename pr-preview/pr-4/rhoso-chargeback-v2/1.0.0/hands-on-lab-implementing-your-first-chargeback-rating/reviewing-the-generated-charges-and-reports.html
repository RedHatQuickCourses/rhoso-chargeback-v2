Reviewing the generated charges and reports

This content is provided in Antora AsciiDoc format.

Reviewing the Generated Charges and Reports

As part of managing your cloud infrastructure costs with RHOSO Chargeback, understanding and verifying the generated charges and reports is paramount. These reports are the culmination of the chargeback process, translating raw resource consumption into meaningful cost insights. They provide the transparency needed for accurate cost recovery and empower tenants to optimize their resource usage.

Understanding Chargeback Reports

RHOSO Chargeback, powered by CloudKitty, acts as the "meter reader" for your OpenStack environment. It takes collected metrics – such as VM uptime or storage consumed – and applies your predefined rating rules to generate structured reports. These reports are typically consumed in JSON format, making them easily machine-readable and integrable with other financial or FinOps solutions.

The primary purpose of these reports is to provide:

  • Transparent Cost Recovery: A clear, itemized breakdown of resource usage and associated costs per tenant or project. This ensures accurate recouping of operational expenses.

  • Trust and Optimization: By exposing how specific consumption (e.g., by project, instance flavor, or metric) translates into costs, tenants gain insights that drive informed decisions, such as optimizing VM configurations or archiving stale data.

The workflow, in essence, is: Set Rating Rules → Collect Metrics → Generate Rating Reports. Reviewing these reports is the final, crucial step to close the loop between infrastructure consumption and financial accountability.

Report Structure and Components

A typical chargeback report generated by CloudKitty provides aggregated data for a specified period and grouping. The output, as demonstrated in the context, is a JSON object containing a summary array. Each element in this array represents a summary of charges for a particular resource type or tenant within the defined timeframe.

Let’s examine the key components of a report entry:

  • tenant_id: Identifies the specific OpenStack tenant (project) responsible for the consumption. This ensures accountability and allows for tenant-specific billing.

  • res_type: Denotes the type of resource being rated, such as ceilometer_cpu, volume, or network. This provides granularity into where costs are incurred.

  • begin: The start timestamp (ISO 8601 format) for the reporting period for which the charges were calculated.

  • end: The end timestamp (ISO 8601 format) for the reporting period.

  • rate: The total calculated cost or units for the res_type within the specified begin and end period, after applying all relevant rating rules. This is the monetary value or consumption unit you’re interested in.

This structured data is ideal for direct ingestion into enterprise financial software, enabling a seamless connection between technical infrastructure usage and business-level cost accountability.

Hands-on Lab: Reviewing Your First Chargeback Report

In this lab, you will generate and review a summary chargeback report using the CloudKitty API. This activity will demonstrate how to retrieve calculated charges and interpret the resulting data.

Prerequisites

  • A deployed RHOSO Chargeback environment with CloudKitty enabled and operational within RHOSP.

  • At least one rating rule already defined and active (as covered in previous lab steps).

  • Some resource consumption data collected by CloudKitty for a period after rating rules were applied.

  • Access to a terminal with curl installed and an authenticated OpenStack tenant token ($TENANT_TOKEN) available as an environment variable.

Steps to Generate and Review a Report

Follow these steps to generate and review a summary chargeback report:

  1. Set your Tenant Token (if not already set) Ensure you have an authenticated OpenStack token for your tenant stored in the $TENANT_TOKEN environment variable. This token is required for authenticating your API request to CloudKitty.

    [source,bash]
----
# Example: Replace with your actual token retrieval method
export TENANT_TOKEN=$(openstack token issue -c id -f value)
----

  2. Generate a Summary Report Use the curl command to query the CloudKitty API for a summary report. You will specify a begin and end timestamp for the desired reporting period, and groupby=res_type to aggregate data by resource type.

    NOTE: The example dates are for illustrative purposes. Adjust the `begin` and `end` parameters to a period where you know resource usage occurred *after* your rating rules were active in your environment.
    [source,bash]
----
curl -X GET \
  -H "X-Auth-Token: $TENANT_TOKEN" \
  "http://localhost:8888/v1/report/summary?begin=2026-02-01T00:00:00&end=2026-03-01T00:00:00&groupby=res_type"
----

    • -X GET: Specifies the HTTP GET method for retrieving data.

    • -H "X-Auth-Token: $TENANT_TOKEN": Passes your authentication token in the request header.

    • "http://localhost:8888/v1/report/summary?begin=2026-02-01T00:00:00&end=2026-03-01T00:00:00&groupby=res_type": This is the API endpoint URL with query parameters:

    • begin: The start date and time for the report.

    • end: The end date and time for the report.

    • groupby=res_type: Instructs CloudKitty to aggregate the charges by resource type. Other groupby options might include tenant_id, project_id, or specific metadata tags depending on your CloudKitty configuration and reporting needs.

  3. Review the Generated JSON Output Upon successful execution, the curl command will return a JSON object containing your chargeback report summary. Examine the output, which will look similar to this:

    [source,json]
----
{
    "summary": [
        {
            "tenant_id": "MoD-project-uuid",
            "res_type": "ceilometer_cpu",
            "begin": "2026-02-01T00:00:00",
            "end": "2026-03-01T00:00:00",
            "rate": 125.50
        },
        {
            "tenant_id": "MoD-project-uuid",
            "res_type": "nova_instance",
            "begin": "2026-02-01T00:00:00",
            "end": "2026-03-01T00:00:00",
            "rate": 200.75
        },
        {
            "tenant_id": "another-project-uuid",
            "res_type": "cinder_volume",
            "begin": "2026-02-01T00:00:00",
            "end": "2026-03-01T00:00:00",
            "rate": 50.00
        }
    ]
}
----

  4. Interpret the Report Data Analyze the summary array:

    • Each object in the array represents a aggregated charge entry.

    • For the first entry, tenant_id: "MoD-project-uuid" utilized ceilometer_cpu resources, incurring a rate (cost) of 125.50 during February 2026. This rate reflects the application of your defined CPU rating rule to the collected CPU usage metrics for that tenant.

    • Similarly, "MoD-project-uuid" also had nova_instance charges of 200.75.

    • The another-project-uuid project incurred 50.00 for cinder_volume usage.

      This report provides a clear, aggregated view of resource consumption costs, enabling you to verify your rating rules and understand the financial impact of different resource types across your tenants.

Conclusion

Reviewing these generated reports is critical for several reasons: it confirms that your chargeback rules are working as expected, provides the necessary data for internal billing or showback, and offers actionable insights for resource optimization. By regularly generating and analyzing these reports, you ensure transparency and maintain control over your cloud spending.
