Setting up the lab environment for Chargeback

In this unit, you will prepare your Red Hat OpenStack Services on OpenShift (RHOSO) environment for chargeback by installing the required dependencies and enabling the Rating service (CloudKitty). You will use Ansible playbooks to automate the setup, while understanding what each step accomplishes based on the concepts covered in the Prerequisites for Rating enablement in RHOSO and How To Enable Rating theory modules.

Connect to the Workstation Machine

All lab tasks are performed from the workstation machine.

Install ansible-core (it might be available already):

[student@workstation ~]$ sudo dnf install ansible-core

You will run all subsequent commands from this machine.

Clone the RHOSO Chargeback and Observability Fundamentals Training Repository

All the automation scripts for this lab are available in a dedicated Git repository. Clone it to your workstation:

git clone https://github.com/opentraining-labs-customizations/rhoso-chargeback.git
cd rhoso-chargeback

The repository contains four directories, each with an Ansible playbook for a specific stage of the lab:

  • pre-deploy/ — Installs Loki, MinIO and COO, required by CloudKitty to access metrics and store the rated data.

  • deploy/ — Updates the OpenStack Control Plane to enable Telemetry, Ceilometer, MetricStorage, and CloudKitty.

  • create-ck-rating/ — Automates the creation of a hashmap rating rule (used in the next unit).

  • validation/ — Validates that the rating configuration was applied correctly (used in the final unit).

Install CloudKitty Dependencies

As covered in the Prerequisites for Rating enablement in RHOSO module, CloudKitty requires a Loki instance for storing rated data, backed by S3-compatible storage. The pre-deploy playbook sets this up for you. It also requires COO, to deploy the MetricStorage instance managed by COO.

Run the playbook:

ansible-playbook pre-deploy/deploy-ck-dependencies.yml

This playbook performs the following actions:

  1. Waits for the OpenShift Route API to become available.

  2. Applies Kubernetes manifests that create:

    • The openshift-operators-redhat and minio-dev namespaces.

    • A Loki Operator subscription (from the stable-6.4 channel).

    • A Cluster Observability Operator subscription (from the stable channel).

    • A MinIO pod with persistent storage (10Gi) for S3-compatible object storage.

    • A Secret named cloudkitty-loki-s3 in the openstack namespace containing MinIO credentials.

  3. Waits for Loki and Cluster Observability Operators to reach Succeeded status (up to 5 minutes).

The playbook references files relative to the repository root. If you are running it from a different directory, override the path with:

ansible-playbook \
  -e "ansible_user_dir=/path/to/rhoso-chargeback" \
  pre-deploy/deploy-ck-dependencies.yml

Enable Telemetry, Ceilometer, MetricStorage, and CloudKitty

With the storage dependencies in place, you can now enable the Rating service itself. As described in the How To Enable Rating module, this involves modifying the OpenStackControlPlane Custom Resource to activate several components.

Run the deploy playbook:

ansible-playbook deploy/update-deployment-for-ck.yml

This playbook performs the following actions:

  1. Adds a CloudKittyPassword entry to the osp-secret Secret.

  2. Activates the Telemetry service in the OpenStackControlPlane CR.

  3. Activates Ceilometer in the OpenStackControlPlane CR.

  4. Activates MetricStorage with pvcStorageClass: nfs-storage.

  5. Activates CloudKitty in the OpenStackControlPlane CR.

  6. Waits for the OpenStackControlPlane to become ready (up to 30 minutes).

This step may take up to 30 minutes to complete while the operator reconciles all changes and deploys the new components. Do not interrupt the playbook while it is waiting.

Verify the Deployment

After the playbook completes successfully, verify that CloudKitty is running.

  1. Check that the CloudKitty pods are in Running state:

    oc get pods -n openstack | grep cloudkitty

    You should see cloudkitty-api and cloudkitty-processor pods running.

  2. Confirm the CloudKitty CR reports a Ready condition:

    oc get -n openstack CloudKitty

  3. Verify that Ceilometer and MetricStorage are also ready:

    oc get ceilometer -n openstack
oc get metricstorage -n openstack

Once all components report a ready state, your environment is fully prepared for rating. In the next unit, you will create your first rating rule.
