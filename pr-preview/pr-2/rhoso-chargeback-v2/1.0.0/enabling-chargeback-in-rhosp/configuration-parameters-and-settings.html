Configuration parameters and settings

Configuration Parameters and Settings for RHOSO Chargeback

This section delves into the essential configuration parameters and settings required to effectively enable and utilize the Chargeback feature within a Red Hat OpenStack Platform (RHOSP) environment with RHOSO. Proper configuration ensures accurate resource consumption tracking and the generation of precise billing reports.

The ability to accurately charge back resource usage hinges on carefully defining which metrics are collected and how they are processed. This involves configuring the underlying observability services to feed the necessary data to the Chargeback (referred to as Cloud Rating in the documentation) component, and then specifically instructing the Chargeback service on how to interpret and rate that data.

Configuring Metrics for Rating Rules

For the RHOSO Chargeback system to function, it needs a continuous stream of relevant resource consumption metrics. As highlighted in the CHAPTER 3. ENABLING CLOUD RATING IN A RHOSO ENVIRONMENT, specifically section 3.3. CONFIGURING METRICS FOR RATING RULES, the foundation of accurate chargeback lies in this initial configuration step.

The "rating rules" are the core logic that transforms raw metric data into billable costs. Before you can define these rules, you must ensure that the metrics they rely upon are properly configured and available.

Key Configuration Parameters for Metrics for Rating Rules:

  • Metric Selection: Identify the specific infrastructure metrics that represent billable resources. This often includes:

    • CPU usage (cores, CPU-hours)

    • Memory consumption (GB, GB-hours)

    • Storage usage (GB, GB-hours for block and object storage)

    • Network I/O (GB transferred, network bandwidth usage)

    • Number of instances/VMs, volumes, or floating IPs These metrics are usually gathered by RHOSO’s observability components (e.g., Prometheus, Ceilometer).

  • Aggregation Methods: Define how collected raw metric data is processed over a billing period. Common aggregation methods include:

    • sum: Total usage over the period (e.g., total GB transferred).

    • average: Average usage over the period (e.g., average CPU utilization).

    • max: Peak usage over the period (e.g., highest memory usage).

    • count: Number of occurrences (e.g., number of VMs provisioned). The choice of aggregation method directly impacts the calculated charge. For instance, billing for average CPU usage differs significantly from billing for peak CPU usage.

  • Granularity: Specify the time interval at which metrics are sampled and processed for rating. Finer granularity (e.g., 5-minute intervals) provides more precise data but requires more storage and processing power, whereas coarser granularity (e.g., hourly) might be less precise but more efficient.

  • Dimensions and Labels: Metrics often come with various dimensions or labels (e.g., project ID, user ID, region, resource type). These are crucial for segmenting costs and attributing them to specific tenants, departments, or projects. Configuration involves ensuring these labels are correctly propagated and can be used in rating rule conditions.

  • Data Retention Policies: While not directly a "rating rule" parameter, ensuring that the observability backend retains metric data for a sufficient period (e.g., 90 days, 1 year) is critical for chargeback reconciliation and historical reporting.

Properly configuring these parameters ensures that the right data, in the right format, is available for the chargeback engine to apply rating rules accurately.

Defining the Rating Service Metrics to Collect

Beyond simply ensuring metrics are available, the Chargeback service (referred to as the Rating Service) needs explicit instructions on which metrics it should actively consume and how to interpret them. This is covered under section 3.4. DEFINING THE RATING SERVICE METRICS TO COLLECT in the context documentation. This step involves configuring the Chargeback service itself to pull or receive the specifically chosen metrics from the observability platform.

Key Configuration Settings for Rating Service Metric Collection:

  • Metric Sources: Configure the Chargeback service to connect to its data sources. This typically involves specifying endpoints for the observability backend (e.g., Prometheus API, OpenStack Gnocchi/Ceilometer endpoints).

  • Collection Schedules/Intervals: Define how often the Rating Service queries or consumes metric data. This often aligns with the desired granularity for chargeback calculations (e.g., once every hour, once a day).

  • Metric Mapping and Transformation: Raw metric names from an observability system might not directly translate to user-friendly or consistent names for chargeback. Configuration often includes mapping rules to standardize metric names (e.g., nova_instance_cpu_usage_seconds_total might map to compute_cpu_usage_hours). In some cases, transformations might be needed, such as converting bytes to GB or seconds to hours.

  • Filtering and Scoping: Configure filters to ensure the Rating Service only collects metrics relevant to the scope of chargeback (e.g., metrics from specific projects, regions, or resource types, excluding infrastructure components not meant for billing).

These settings bridge the gap between the raw data generated by your RHOSP environment and the structured input required by the Chargeback engine.

Hands-on Lab: Configuring Core Chargeback Parameters (Conceptual)

While the specific commands and user interface steps can vary based on the exact RHOSO version and deployment specifics, this conceptual lab activity walks through the general process of configuring the parameters discussed. This activity focuses on understanding what needs to be configured, rather than providing prescriptive syntax not explicitly detailed in the current context.

Scenario:

You need to ensure that the RHOSO Chargeback system is configured to track and rate CPU usage and storage consumption for all tenant VMs in your OpenStack environment, aggregating usage hourly.

Steps:

  1. Identify Required Metrics:

    • Action: Determine the precise metric names that RHOSO’s observability components provide for CPU usage and storage.

    • Example (Conceptual): You would look for metrics like openstack_nova_instance_cpu_utilization and openstack_cinder_volume_size_bytes_total.

  2. Configure Observability for Chargeback:

    • Action: Verify that your RHOSO observability setup is collecting these identified metrics and retaining them for the required billing period. This may involve adjusting Prometheus scrape configurations or Ceilometer pipeline definitions.

    • Expert Insight: Ensure that labels like project_id, tenant_id, and resource_id are consistently attached to these metrics, as they are crucial for accurate cost allocation.

  3. Define Rating Service Metric Collection:

    • Action: Instruct the RHOSO Chargeback (Rating) service to actively collect and process these metrics. This is typically done through configuration files or API calls to the Chargeback service.

    • Example (Conceptual Configuration Snippet - YAML representation):

# This is a conceptual representation based on common configuration patterns.
# Actual syntax will vary based on RHOSO Chargeback implementation and API versions.
rating_service:
  metrics_collection:
    - name: 'cpu_usage_hours'
      source_metric: 'openstack_nova_instance_cpu_utilization'
      aggregation: 'average'
      granularity: '1h'
      dimensions: ['project_id', 'vm_id']
      transformations:
        - type: 'unit_conversion'
          from: 'percent_per_second'
          to: 'cpu_hours' # (e.g., average % * time_period / 100)
    - name: 'storage_gb_hours'
      source_metric: 'openstack_cinder_volume_size_bytes_total'
      aggregation: 'average'
      granularity: '1h'
      dimensions: ['project_id', 'volume_id']
      transformations:
        - type: 'unit_conversion'
          from: 'bytes'
          to: 'gb'
      filters:
        - label: 'volume_type'
          value: 'standard' # Only collect standard storage for this rule
  data_sources:
    - type: 'prometheus'
      endpoint: 'http://rhoso-prometheus-server:9090'
      # Add authentication details as required for your environment

  1. Apply and Verify:

    • Action: Deploy or reload the Chargeback service configuration to activate your new settings.

    • Verification: Monitor the Chargeback service logs to ensure it successfully connects to the metric sources and starts processing the defined metrics without errors. You would typically look for log entries indicating successful metric ingestion and processing.

This conceptual lab highlights the critical thinking and planning required to map your resource consumption to billable metrics through precise configuration. The specific implementation details will be guided by the official RHOSO Chargeback documentation and API specifications.
