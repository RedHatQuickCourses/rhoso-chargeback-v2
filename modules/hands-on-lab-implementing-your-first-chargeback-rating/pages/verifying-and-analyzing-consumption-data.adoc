# Verifying and analyzing consumption data

In this final unit, you will validate that your rating configuration from the previous unit was applied correctly. You will use the validation playbook from the lab repository to run a series of automated checks against your RHOSO environment.

== Run the Validation Playbook

The validation playbook checks each component of the hashmap rating configuration you created and reports pass or fail for each one.

Run the playbook:

[source,bash]
----
ansible-playbook validation/validate-ck-rating.yml
----

If you used non-default values when creating your rating rule (for example, a different flavor or cost), pass them as variables so the validation checks against the correct values:

[source,bash]
----
ansible-playbook validation/validate-ck-rating.yml \
  -e flavor_name=m1.small \
  -e rating_cost=0.5
----

== Interpreting the Results

The playbook performs the following checks in order:

[cols="1,2"]
|===
| Check | What It Validates

| Hashmap module enabled
| The `hashmap` rating module is active. Fails if the module is missing or disabled.

| Group exists
| The group `instance_uptime_flavor_id` exists in the hashmap configuration.

| Service exists
| The service `instance` is registered under the hashmap module.

| Field exists
| The field `flavor_id` is defined under the `instance` service.

| Mapping exists
| A mapping with the expected type and group association is present.

| Rating data flowing
| Rated data is being generated by CloudKitty. This is a soft check -- it warns but does not fail if no data is available yet.
|===

If all checks pass, your rating configuration is correctly set up and CloudKitty is processing your rules.

[NOTE]
====
The "rating data flowing" check may show a warning if you run the validation shortly after creating the rating rule. CloudKitty processes metrics on a periodic cycle, so it may take a few minutes before rated data appears. This warning does not indicate a configuration problem.
====

== Generate Rating Reports

After validating your configuration, you can query CloudKitty for the rated data it has produced. CloudKitty provides two levels of detail: **summaries** for aggregated totals and **dataframes** for individual rated data points. Think of summaries as the monthly statement and dataframes as the individual line items on the receipt.

=== Summaries

The `openstack rating summary get` command returns aggregated charges for a given time period. As a tenant, you can only see your own project's data -- CloudKitty enforces tenant-aware access control automatically.

Retrieve a summary for a specific time range:

[source,bash]
----
oc exec -n openstack openstackclient -- \
  openstack rating summary get --begin 2026-02-01 --end 2026-03-01
----

The output shows the total rated cost for each resource type within the specified period. Key fields to look for:

* `Tenant ID` -- The project responsible for the consumption.
* `Resource Type` -- The type of resource being rated (e.g., `instance`).
* `Rate` -- The total calculated cost for that resource type within the time window.

[TIP]
====
If the output is empty, CloudKitty may not have completed a processing cycle yet. The processor collects and rates metrics every hour by default, so wait at least one hour after creating your rating rule and try again.
====

=== Dataframes

While summaries give you the big picture, dataframes show the individual rated data points stored in Loki. Use them to verify that the correct mapping was applied to each resource, or to debug a missing metric.

Retrieve dataframes for a narrow time window (e.g., one hour):

[source,bash]
----
oc exec -n openstack openstackclient -- \
  openstack rating dataframes get \
  --begin 2026-03-01T00:00:00Z --end 2026-03-01T01:00:00Z
----

Each row in the output represents a single rated data point. The key columns are:

* `Begin` / `End` -- The collection window. CloudKitty produces one-hour windows by default.
* `Metric Type` -- The metric name as configured in `metrics.yaml` (e.g., `ceilometer_cpu`).
* `Qty` -- The raw quantity after any transformations have been applied. For instance metrics using `NUMBOOL`, this equals `1` when the instance was running during the window.
* `Price` -- The rated value after hashmap rules are applied. This is where you verify that your mapping cost (e.g., `0.3` for the `tiny` flavor) was correctly applied.
* `Group By` -- Label values that enable drill-down, such as `flavor_id`, `flavor_name`, `project`, and `resource` (the VM UUID).

By comparing the `Price` column against the cost you defined in your mapping, you can trace the full path from raw metric to final price and confirm that your rating rule is working as expected.

== Troubleshooting

If any of the validation checks fail, follow these steps to diagnose and fix the issue:

. **Review the failed check**: The playbook output identifies which specific component is missing or misconfigured.

. **Re-check Unit 2 steps**: Go back to the previous unit and verify that each command completed successfully. Common issues include:
* Forgetting to replace placeholder IDs (`<service_id>`, `<field_id>`, `<group_id>`) with actual values.
* Typos in the flavor name or field name.
* Skipping the hashmap module enablement step.

. **Use the cleanup and recreate playbooks**: If you want to start fresh, clean up the existing configuration and recreate it automatically:
+
[source,bash]
----
ansible-playbook create-ck-rating/cleanup-ck-rating.yml
ansible-playbook create-ck-rating/create-ck-rating.yml
----
+
Then re-run the validation playbook.

. **Check CloudKitty pod logs**: If the configuration looks correct but data is not flowing, inspect the CloudKitty processor logs for errors:
+
[source,bash]
----
oc logs -n openstack $(oc get pods -n openstack -l app=cloudkitty -o name | head -1)
----

For additional guidance on diagnosing issues with metric collection, Ceilometer, MetricStorage, or CloudKitty, refer to the xref:basic-rating-troubleshooting:troubleshooting-cloudkitty-deployment.adoc[Troubleshooting CloudKitty Deployment] and xref:basic-rating-troubleshooting:troubleshooting-rating-and-cloudkitty-configuration.adoc[Troubleshooting Rating and CloudKitty Configuration] modules.
